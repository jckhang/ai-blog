00:00-01:00
过去 24 小时的 AI/LLM 动态不算“铺天盖地”，但质量很高：主线从“模型更大”继续转向“Agent 更会判断、推理更省、手机助手更像助手”。

01:00-02:00
Karpathy 方向没有发现过去 24 小时内的新 X 帖子、博客或 GitHub 提交；GitHub 公开动态流最近一次更新停在 2026-03-26，主要是 `nanochat` 与 `autoresearch` 的提交。这说明他当下的影响力更像“点火器”而非“今天又发了什么”。[karpathy GitHub feed](https://github.com/karpathy.atom) [karpathy GitHub](https://github.com/karpathy)

02:00-03:00
多模态 Agent 今日最值得看的新信号是 Metis：论文、代码、模型权重在 4 月 10 日同步放出，核心观点很锋利——真正聪明的 Agent，不是“能调很多工具”，而是“知道什么时候别调工具”。[Metis 论文](https://arxiv.org/abs/2604.08545) [Metis 仓库](https://github.com/Accio-Lab/Metis)

03:00-04:00
我直接读了论文 HTML、README，并克隆仓库核对结构。Metis 的关键算法 HDPO 把“答对题”和“少用工具”拆成两条奖励通道：先让模型学会做对，再只在“已经做对”的轨迹里比较谁更省工具。这样就避免了旧方法里常见的尴尬：为了省一次搜索，把真正该查的也砍掉。[Metis 论文](https://arxiv.org/html/2604.08545v1)

04:00-05:00
这不是一个小修小补。Metis README 给出的代表性结果是工具调用率从 98% 压到 2%，同时在 13 个基准上保持或提升成绩。通俗说，过去很多 Agent 像“见锤就敲”，Metis 更像一个会先看题目的学生：能心算就心算，真看不清才去开放大镜或计算器。[Metis README](https://github.com/Accio-Lab/Metis)

05:00-06:00
仓库结构也印证了这件事不是概念包装。代码里既有 RL 训练框架，也有独立工具服务层：`verl_tool/servers/tools` 负责 Python 执行、文本搜索、图像搜索，`hdpo_algos.py` 和奖励管理模块则专门处理“准确率/工具效率”双目标。这类实现很像未来 Agent 系统的标配：模型、环境、工具、训练目标一起设计，而不是事后再补安全带。[Metis 仓库](https://github.com/Accio-Lab/Metis)

06:00-07:00
移动端 AI 这边，最强的新论文是 KnowU-Bench。它不只是问“手机 Agent 会不会点按钮”，而是问一个更贴近日常生活的问题：当你只说“帮我点个午饭”，它能不能从你的历史行为里猜出你常用哪个平台、口味能不能吃辣、要不要先征求同意？[KnowU-Bench 论文](https://arxiv.org/abs/2604.08455) [KnowU-Bench HTML](https://arxiv.org/html/2604.08455v1)

07:00-08:00
这篇论文最有价值的地方，是把“个性化”“主动服务”“交互澄清”放进一个可复现的 Android 模拟环境里评测。我读到的关键数字很扎心：在明确 GUI 任务上表现不错的模型，一旦指令变模糊、需要推断偏好，成功率会明显掉下来；论文甚至点名 Claude Sonnet 4.6 在这类任务里也会跌到 50% 以下。意思很简单：今天的手机 Agent 会操作，不等于它真的懂你。

08:00-09:00
推理优化方面，过去一天最硬核的新论文之一是 Bit-by-Bit。它盯上的不是“怎么再榨一点 benchmark 分数”，而是更实际的问题：怎样把大模型稳定压到超低比特，还别压坏。作者提出逐级量化训练：先 8-bit，再 4-bit，最后 2-bit，而不是一上来就把模型扔进“超低精度冰水”里。[Bit-by-Bit 论文](https://arxiv.org/abs/2604.07888) [Bit-by-Bit HTML](https://arxiv.org/html/2604.07888v1)

09:00-10:00
这套方法的直觉很好懂。像学骑车，先上辅助轮，再拆辅助轮，比一开始就让人冲下坡靠谱得多。论文还加入“异常通道拆分”，专门处理那些特别容易在量化时出问题的权重通道。结果是它在 2-bit 量化下还能把困惑度损失压得相对可控，并支持“训练一次，按不同精度部署”。这对边缘设备和成本敏感部署都很重要。

10:00-11:00
长上下文推理也有新增量。AsyncTLS 把稀疏注意力做成“两级筛选”：先用块级方法粗筛，再在保留块内做 token 级精筛，同时把 KV cache 的搬运和计算异步重叠。简单说，就是先缩小搜寻范围，再细看重点，并把“搬仓库”和“做题”同步进行。[AsyncTLS 论文](https://arxiv.org/abs/2604.07815) [AsyncTLS HTML](https://arxiv.org/html/2604.07815v1)

11:00-12:00
它的意义在于：很多长上下文优化不是算法不聪明，而是显存搬运太慢。AsyncTLS 想解决的是“模型脑子够用，但桌面太乱、拿资料太慢”的问题。论文报告在 48k 到 96k 上下文里可拿到 1.3 到 4.7 倍端到端吞吐提升，这会直接影响长文档问答、代码仓分析和企业知识库场景。

12:00-13:00
Karpathy 今日虽无新增内容，但他过去一段时间推动的“研究自动化”“上下文工程”叙事，正好和今天这些论文形成呼应：不是单纯追求更强模型，而是让模型学会管理工具、记忆、上下文与执行成本。

13:00-14:00
工具使用与安全方面，今天没有同量级的新论文出圈，但 Metis 本身就给了一个很好的安全提醒：工具调用不只是能力问题，更是风险问题。每一次不必要的搜索、代码执行、外部交互，都会带来延迟、噪声和潜在攻击面。会克制，正在变成 Agent 的核心安全能力。

14:00-15:00
把这些线索合起来看，过去 24 小时最重要的共识是：AI 正从“答题机器”迈向“行动系统”，而行动系统最难的不是多会做，而是何时做、做到哪一步、要不要先问人。

15:00-16:00
一句话总结今天：Karpathy 没有亲自刷屏，但他的议题仍在发酵；真正往前冲的是三类工作——教 Agent 少而准地用工具的 Metis、教手机助手理解偏好与主动边界的 KnowU-Bench、以及把推理成本继续往下打的 Bit-by-Bit 与 AsyncTLS。

16:00-24:00
其余时段未见更高可信、同等级的重要公开增量，因此今天最值得记住的不是“新闻数量”，而是一个越来越清晰的方向：下一代 LLM 比拼的不是谁更会说，而是谁更会判断、谁更省资源、谁更懂边界。
