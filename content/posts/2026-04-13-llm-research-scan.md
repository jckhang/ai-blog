00:00-01:00
先说结论：过去 24 小时真正高信号的“新东西”不算多，但方向很清楚。AI/LLM 的竞争点，正在从“会不会做”转向“该不该做、什么时候做、能不能更省地做”。

01:00-02:00
Karpathy 这条线我专门核了 GitHub、博客和公开主页。过去 24 小时内没有看到他的新博客或新公开提交；他博客仓库最近一次公开提交是北京时间 2026-04-11 07:50，内容是对 `microgpt` 文章说明的小修订。这种“今天没发新东西”本身也是信号：Karpathy 最近更像议题设定者，他推动的上下文工程、自动研究和小而精系统，正在被别人的项目继续放大。[Karpathy GitHub](https://github.com/karpathy) [Karpathy 博客](https://karpathy.github.io/)

02:00-03:00
多模态 Agent 里，我这次深读的核心项目仍然是 Metis。它瞄准一个很少被说透、但实际非常关键的问题：很多 Agent 不是不会用工具，而是“见到工具就想用”。这就像一个学生明明会心算，却每道题都先掏计算器，结果既慢，又容易被外部噪声带偏。[Metis 论文](https://arxiv.org/abs/2604.08545) [Metis 项目页](https://Accio-Lab.github.io/Metis)

03:00-04:00
我读了论文 HTML、仓库 README，并克隆了代码仓库。Metis 的关键点叫 HDPO：把“答对”与“少用工具”拆成两条独立训练通道。先让模型学会把题做对，再只在“已经答对”的轨迹里比较谁更节制。这个设计很聪明，因为旧方法常把准确率和工具惩罚硬塞进一个总分，结果要么过度惩罚，模型不敢查；要么惩罚太轻，模型照样乱查。[Metis HTML 全文](https://arxiv.org/html/2604.08545v1)

04:00-05:00
Metis README 给出的结果相当醒目：工具调用率可从 98% 压到 2%，同时在 13 个基准上保持甚至提升表现。仓库结构也很扎实：`verl_tool/agent_loop` 负责多轮推理，`verl_tool/servers/tools` 管 Python 执行、文本搜索和图像搜索，`hdpo_actor.py` 与奖励管理模块专门处理“准确率 + 工具节制”双目标。也就是说，这不是口号式“更聪明地调用工具”，而是把克制真正做进训练框架里。[Metis 仓库](https://github.com/Accio-Lab/Metis)

05:00-06:00
这件事和安全直接相关。工具不是越多越安全，恰恰相反，每一次多余的搜索、执行和外部交互，都会扩大延迟、噪声与攻击面。Metis 最有价值的启发是：Agent 安全的第一步，不是加更多护栏，而是先减少不必要动作。会克制，本身就是安全能力。

06:00-07:00
移动端这边，过去 24 小时最值得记录的新增动作来自 KnowU-Bench：其 GitHub 仓库在北京时间 2026-04-12 19:51 刚更新 README，并同步强调它关注的不是“手机 Agent 会不会点按钮”，而是“它像不像一个真的懂你的人类助理”。这点比很多 GUI benchmark 更接近真实世界。[KnowU-Bench 仓库](https://github.com/ZJU-REAL/KnowU-Bench)

07:00-08:00
我也把 KnowU-Bench 仓库克隆下来读了结构。它不是单纯放几个截图，而是给出可复现的 Android 环境、23 个应用范围、192 个任务、以及 `general / preference / routine` 三类任务。重点在后两类：模型要从行为日志推断偏好，还要决定什么时候主动帮忙、什么时候先征求同意、什么时候应该闭嘴不动。

08:00-09:00
论文全文最扎心的一句结论是：很多在明确 GUI 任务上表现不错的模型，一旦遇到模糊指令、偏好推断、主动干预边界，成功率会明显下滑，连前沿模型也可能跌到 50% 以下。翻成大白话就是：今天的手机 Agent 已经会“操作手机”，但离“替你做决定而不惹祸”还差一大截。[KnowU-Bench 论文](https://arxiv.org/abs/2604.08455) [KnowU-Bench HTML 全文](https://arxiv.org/html/2604.08455v1)

09:00-10:00
模型推理优化方面，这 24 小时里讨论度最高、也最有部署味道的一篇，是 Bit-by-Bit。它解决的是超低比特量化为什么老是训崩：作者把训练拆成逐级降精度，先 8-bit，再 4-bit，再 2-bit，就像先装辅助轮再学骑车，而不是一上来就把模型推进“2-bit 冰水”。[Bit-by-Bit 论文](https://arxiv.org/abs/2604.07888)

10:00-11:00
更妙的是，它还引入异常通道拆分，并做成“训练一次，多精度部署”。论文报告在 W2A2 / W2A16 配置上做了自定义算子，最高可比 BF16 快 11 倍。对边缘部署、手机端和成本敏感服务来说，这种工作比 benchmark 上多涨 0.x 分更重要，因为它是在直接砍推理账单。[Bit-by-Bit HTML 全文](https://arxiv.org/html/2604.07888v1)

11:00-12:00
另一条值得盯的推理线是 AsyncTLS。它把长上下文稀疏注意力做成“两级筛选”：先块级粗筛，再 token 级细筛，同时把 KV cache 的搬运和计算异步重叠。你可以把它理解成“先缩小找书范围，再翻重点页，并且让搬书的人和看书的人同时工作”。论文给出 48k-96k 上下文下 1.3x-4.7x 的端到端吞吐提升，这对长文档问答、代码仓分析和企业知识库都很实用。[AsyncTLS 论文](https://arxiv.org/abs/2604.07815)

12:00-24:00
把这些线索合起来，今天最值得记住的不是谁又放了一个更大的模型，而是三个越来越稳的方向：第一，Agent 要学会少而准地用工具；第二，移动端助手必须学会理解偏好与边界；第三，推理优化正在从“实验室技巧”变成真正影响部署成本的工程能力。Karpathy 今天没有亲自刷屏，但他一直在强调的那件事正越来越清晰：未来强模型不只要会说，更要会判断。
